# Рішення та пояснення

## Захист від дублювання

Для захисту від дублювання заявок обрано **Redis** з TTL 10 хвилин. Ключ формується як MD5-хеш комбінації `phone + address`.

**Чому саме Redis:**
- Швидкий доступ O(1) для перевірки існування ключа
- Вбудований механізм TTL — ключ автоматично видаляється через 10 хвилин
- Не навантажує основну базу даних додатковими запитами
- Атомарність операцій запобігає race conditions

**Альтернативи, які було розглянуто:**
- Перевірка через PostgreSQL (`SELECT ... WHERE created_at > now() - interval '10 min'`) — працює, але повільніше та створює додаткове навантаження на БД
- Унікальні constraint'и в БД — не підходять, бо обмеження тимчасове (10 хв)

## Найслабше місце реалізації

Найслабшим місцем є відсутність retry-механізму для Telegram-повідомлень. Якщо Telegram API недоступний у момент відправки, повідомлення буде втрачено. Для production-рішення варто додати:
- Retry з exponential backoff в Celery-задачах
- Dead letter queue для повідомлень, що не вдалося відправити
- Моніторинг та алерти

## Що б доробив за 2 додаткові години

1. **Тести** — unit-тести для API, інтеграційні тести для Celery-задач та Redis-дедуплікації
2. **Swagger/OpenAPI** — автоматична документація API через drf-spectacular
3. **Логування** — структуроване логування з налаштуванням рівнів
4. **Retry для Celery** — автоматичний повтор невдалих задач з backoff
5. **Валідація телефону** — нормалізація формату номера перед дедуплікацією
